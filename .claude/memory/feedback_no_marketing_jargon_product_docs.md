Rule: All product, strategy, roadmap, feature, and pitch docs written for Vishal must be in plain English. No marketing phrases. No tech jargon. No landing-page copy.

**Why:** Vishal explicitly asked for this on 2026-05-13 during the Creator-SaaS-Tool-by-KnowAI project. He read the strategy doc and roadmap.html and said: "I need clear, what the app does, how it does it, all the features it have, no tech jargon or marketing." He has [[feedback_slide_plain_language]] for slides already — this is the broader version: applies to ALL written product/strategy docs, not just slides. He reads docs to understand and decide, not to be sold to.

**How to apply:**

Avoid these patterns:
- "Defensible wedges" / "moats" / "white space" / "TAM"
- "AI-driven [X] engine" / "[X] OS" / "[X] platform" as identifiers
- "Pillars" / "differentiators" / "value props" / "hero arc"
- "All-in-one" / "next-gen" / "best-in-class" / "world-class"
- "Loop" / "stack" / "rail" / "brain" / "lynchpin" as nouns
- Hero copy that sounds like a landing page

Prefer these patterns:
- "What it does, how it does it, what's in it" beats "vision, pillars, differentiators"
- "It's an app that does X for Y" beats "the [X] [OS/engine/platform] for [Y]"
- Plain function names: "connects to" instead of "integration layer"; "writes drafts" instead of "AI-driven generation"; "the list of work" instead of "the pipeline"
- Step-by-step "the user opens the app and..." narrative beats "the system orchestrates..."
- Keep proper-noun names of real things (Instagram Trial Reels, Beehiiv, gridd.ai, IG, TikTok stay — they're the actual names)

When to relax:
- Visual mockups in HTML can stay stylized (the words in them still plain)
- Brand identity copy (the company name, taglines) is a separate exercise — only when explicitly working on branding
- Code/config files don't get rewritten
