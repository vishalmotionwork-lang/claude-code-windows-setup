When Vishal asks to "turn this into a board / SOP / make it usable," and it started from a competitor's reel or video:

- He does NOT want a video breakdown (script, frame-by-frame, funnel analysis, attribution to the creator) once it moves into "build it for us" territory.
- He does NOT want their screenshots/frames copy-pasted/embedded. "Build it on our own" = recreate the visuals natively (e.g. story-slide cards drawn as FigJam frames), not embed their images.
- He wants the framework STRIPPED to: **what it is** (1–2 lines) + **how to use it** (steps + the actual templates). Concise.
- He values being asked detailed clarifying questions before a rebuild (generic vs niche-tailored, depth per item, keep/drop the strategy layer, where to build).

**Why:** A teardown explains someone else's content; he wants a reusable tool he can act on. Embedded screenshots read as lazy/ugly ("you just copy-pasted the images… looks like shit").

**Concrete representation that landed (2026-06-09):** for a story-sequence / multi-slide framework, draw each slide as a real **9:16 story mockup** — dark accent-tinted frame, 5-segment story progress bar (current segment lit), role tag, the ACTUAL slide copy centered like story text, an IG "Send message" bar; the how-to prompt goes as a small caption BENEATH the slide, not on it. "Show the framework really good" = make it look like the real artifact, not a brief.

**How to apply:** For "make it usable," default to a clean native rebuild of just the framework + how-to. Save the teardown (the [[reference_competitor_teardown_workflow]] HTML) as a separate analysis artifact, don't merge it into the usable board. Native FigJam UI cards (see [[reference_figma_image_embed_cors]] designed-UI note) over embedded frames. Ask scope questions first. Generic-reusable examples use bracketed placeholders, never fabricated specifics ([[feedback_no_script_fabrication]]).
